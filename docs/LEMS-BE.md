## **1. Introduction & Architecture Overview**

This document outlines the backend implementation strategy for LEMS. The backend will be built as a highly modular API and background task runner using **NestJS** (Node.js/TypeScript). It will interface with a **PostgreSQL** database (via Prisma or TypeORM) and the **Solana Blockchain** (via `@solana/web3.js` and WebSockets).

### **1.1. Core Backend Principles**

- **Modular Design:** Leveraging NestJS modules to cleanly separate HTTP controllers (API), scheduled tasks (Cron), and event-driven listeners (WebSockets).
- **Zero-Trust RBAC:** Every endpoint must strictly verify the user's role via standard NestJS Guards before executing logic or querying the database.
- **Idempotency & Replay Protection:** Blockchain events can be unpredictable. The database schema and ingestion logic must guarantee that no transaction signature is processed or accounted for twice.
- **Resilient Blockchain Syncing:** The system relies on WebSocket push events (`onLogs`) for the 10-second real-time requirement, backed by a robust polling fallback (Back-filler) for RPC outages.

## **2. Authentication & Role-Based Logic**

The backend relies on cryptographic proofs rather than passwords. It utilizes a stateless JWT architecture for session management.

### **2.1. Signature Verification Flow**

- **Module:** `AuthModule`
- **Logic:**
  1. The frontend submits a payload containing the `wallet_address`, a plaintext `message`, and the cryptographic `signature`.
  2. The `AuthService` utilizes the `ed25519` curve (via `tweetnacl` or `@solana/web3.js`) to verify that the signature was generated by the provided wallet address.
  3. If valid, the service queries the `users` and `roles` tables to fetch the user's assigned role (`ADMIN` or `USER`).
  4. The service generates a JWT containing the `sub` (user UUID), `wallet_address`, and `role`.

### **2.2. Guard Protection (RBAC)**

- **`JwtAuthGuard`**: Ensures the request contains a valid JWT. Applied globally to protected routes.
- **`RolesGuard`**: A custom NestJS Guard using the `@Roles('ADMIN')` decorator.
  - If a `USER` attempts to access an `@Roles('ADMIN')` route (e.g., `/api/v1/registry`), the guard immediately throws a `403 Forbidden` exception.
  - For shared routes like `GET /api/v1/transactions`, the controller checks the `req.user.role`. If `USER`, it forcibly overrides the `pm_id` query parameter with `req.user.id` to scope the database query.

## **3. Module & Service Implementations**

### **3.1. Registry Module (`RegistryModule`)**

Manages the core data of the application. Accessible only by Admins.

- **`RegistryController`**: Exposes standard CRUD operations (`POST`, `GET`, `PATCH`).
- **`RegistryService`**: Handles business logic, such as ensuring a newly added PM's `wallet_address` is exactly 44 characters and is a valid Base58 public key. Handles the "Emergency Pause" logic by toggling `is_active` in the DB.

### **3.2. Blockchain Monitor Module (`BlockchainModule`)**

The engine behind the real-time `<GlobalTransactionFeed />`.

- **`WalletMonitorService`**:
  - On application start, queries all `active` PM wallets from the DB.
  - Opens a WebSocket connection to the Solana RPC using `connection.onLogs()`.
  - When an event fires, it passes the transaction signature to the `MetadataExtractorService`.
- **`MetadataExtractorService`**:
  - Fetches the parsed transaction using `getParsedTransaction`.
  - Identifies if it's a `SPEND` or `REFILL`.
  - Extracts the merchant name from the instruction data.
  - Persists the record to the `transactions` table (relying on the unique constraint on the `signature` column to prevent duplicates).
- **`TransactionBackfillerService`**:
  - Runs on startup and periodically (e.g., every 15 mins). Queries `getSignaturesForAddress` to catch any transactions missed during RPC disconnects or deployments.

### **3.3. Refill Engine Module (`RefillModule`)**

Automates the 1st-of-the-month distributions.

- **`RefillCronService`**: Uses `@nestjs/schedule` (`@Cron('0 0 1 * *')`). Iterates through all `is_active = true` users, calculates `target_balance - current_balance`, and stages a "Batch Refill Proposal" in memory/DB.
- **`ExecutionService`**:
  - Triggered by the Admin via the `POST /api/v1/refill/propose` endpoint.
  - Injects the Master Treasury private key from secure environment variables.
  - Appends a `setComputeUnitPrice` instruction (Dynamic Priority Fees) based on current network congestion.
  - Signs and broadcasts the batch transaction to the Solana network.

## **4. Custom Interceptors & Background Tasks**

To keep the controllers clean and maintain strict accounting compliance, the backend utilizes NestJS interceptors and exception filters.

| **Component Name**            | **Purpose & Internal Logic**                                                                                                                                                                                           |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `AuditLogInterceptor`         | Intercepts all successful `POST`/`PATCH` requests to Admin endpoints. Extracts the Admin's `wallet_address` from the JWT and writes a record to an `audit_logs` table (Action, Timestamp, Target).                     |
| `PrismaClientExceptionFilter` | Catches database-level errors (like `UniqueConstraintViolation` on a transaction signature) and maps them to appropriate HTTP responses or suppresses them silently for the `BlockchainMonitorService`.                |
| `WSEventEmitterInterceptor`   | An interceptor tied to the `transactions` table insertion. Whenever a new transaction is successfully saved, it triggers the NestJS Gateway to emit a `NEW_TRANSACTION` WebSocket event to connected frontend clients. |

## **6. API Endpoints (Controllers)**

Data Transfer Objects (DTOs) with `class-validator` will strictly enforce request payloads.

### **6.1. Auth Controller**

**`POST /api/v1/auth/verify`**

- **Access:** Public
- **Body DTO (`VerifySignatureDto`):**
  - `wallet_address`: string (Base58)
  - `message`: string
  - `signature`: string
- **Response:** JWT Token and User Profile.

### **6.2. Registry Controller**

**`GET /api/v1/registry`**

- **Access:** `@Roles('ADMIN')`
- **Response:** Array of PM objects (id, name, project_id, wallet, target, status).

**`POST /api/v1/registry`**

- **Access:** `@Roles('ADMIN')`
- **Body DTO (`CreatePMDto`):**
  - `name`: string (`@IsString()`, `@IsNotEmpty()`)
  - `wallet_address`: string (`@IsSolanaAddress()`)
  - `target_balance`: number (`@IsPositive()`)
  - `project_id`: string
  - `currency_id`: string
- **Effect:** Creates user, triggers `WalletMonitorService` to subscribe to the new address.

### **6.3. Transaction Controller**

**`GET /api/v1/transactions`**

- **Access:** `@UseGuards(JwtAuthGuard)` (Both Admin and User)
- **Query DTO (`TransactionQueryDto`):**
  - `limit`: number
  - `type`: enum (SPEND/REFILL)
  - `pm_id`: string (Optional for Admin, ignored and overridden for Users)
- **Response:** Paginated transaction list.

### **6.4. Refill Controller**

**`POST /api/v1/refill/propose`** (Execution)

- **Access:** `@Roles('ADMIN')`
- **Body DTO:** `{ "reason": string }`
- **Effect:** Executes the batch transfer using the Master Treasury key. Returns `batch_signature`.

## **7. WebSocket Gateway (Real-Time Feed)**

The backend uses `@nestjs/websockets` to push data down to the React frontend.

### **7.1. Event: `NEW_TRANSACTION`**

- **Trigger:** Fired by the `WSEventEmitterInterceptor` when the `MetadataExtractorService` saves a new record.
- **Payload:** The full serialized transaction object.
- **Room Routing:** - Admins are subscribed to the `global_feed` room.
  - PMs are subscribed to a specific `pm_feed_{uuid}` room. The Gateway strictly routes the event based on the user's validated JWT during socket connection.

### **7.2. Event: `PM_STATUS_CHANGED`**

- **Trigger:** Fired when an Admin toggles the "Pause" button (`PATCH /api/v1/registry/:id/status`).
- **Payload:** `{ pm_id: "uuid", is_active: false }`
- **Effect:** Notifies all connected Admins to update their registry UI instantly.

## **8. Error Handling & Validation States**

Because the API triggers real-world financial movements, payload validation must be impenetrable.

### **8.1. DTO Validation (`class-validator`)**

- We will implement a custom `@IsSolanaAddress()` decorator that attempts to decode the provided string using `@solana/web3.js` `PublicKey` constructor. If it throws, the API automatically returns a `400 Bad Request`.
- **Target Balance:** Validated using `@IsNumber()` and `@Min(1)`. Cannot be negative or zero.

### **8.2. Treasury Fund Safeguards**

- If `POST /api/v1/refill/propose` is called, the `ExecutionService` must simulate the transaction (`simulateTransaction` via RPC) before signing. If the simulation fails (e.g., insufficient master treasury funds), it aborts and returns a `422 Unprocessable Entity` with the simulation error, preventing a failed on-chain transaction fee.

## **9. Non-Functional Requirements (NFRs) & Tech Stack Standards**

### **9.1. Backend Ecosystem**

- **Framework:** NestJS (Express under the hood).
- **ORM:** Prisma (Preferred for strict TypeScript typing mapping directly to the relational DB schema).
- **Blockchain SDK:** `@solana/web3.js` (Core RPC interactions).

### **9.2. Security & Environment Management**

- **Master Key Security:** The Master Treasury private key is **never** stored in the database. It exists strictly as a Base58 encoded string in the `.env` file (or AWS Secrets Manager / Google Secret Manager). The `ExecutionService` loads it temporarily into memory and immediately allows the garbage collector to clear it after the signature is generated.
- **CORS:** Strictly limited to the specific domains hosting the LimeChain frontend to prevent cross-origin attacks.

### **9.3. RPC Resilience & Performance**

- **RPC Fallback Array:** The `.env` will contain an array of RPC endpoints (e.g., Helius, QuickNode, Alchemy). The `BlockchainModule` will implement a round-robin or fallback strategy if `connection.onLogs` drops or a rate limit (`429`) is hit.
- **Database Indexing:** The `pm_id` and `block_time` columns in the `transactions` table will be heavily indexed to ensure the `GET /api/v1/transactions` endpoint responds in under 50ms, even with hundreds of thousands of historical card swipes.
